Archetype is developed as a **git superproject** with each component wired in as a **git submodule**. This page gets you from a fresh clone to a running local stack, and shows where everything lives. Once you are set up, read [Working in the codebase](/contribute/architecture/) and the [Contribution workflow](/contribute/workflow/).

## Repository layout

The workspace root is a superproject that tracks pointers to four component repos plus root-level docs:

```text
archetype/                 # superproject (pointers + docs)
├── api/                   # Django backend        (submodule)
├── frontend/              # Next.js frontend      (submodule)
├── infrastructure/        # Docker Compose, nginx, certs (submodule)
├── site-gh/               # docs site             (submodule)
├── legacy/                # read-only DigiPal source (reference only)
├── CLAUDE.md
└── ROADMAP.md
```

Each submodule is its own repository with its own CI and pre-commit hooks. `legacy/` is kept for behavioural reference during the rebuild — do not modify it.

## Clone and initialise submodules

Clone the superproject and pull every submodule in one step:

```bash
git clone --recurse-submodules <superproject-url> archetype
cd archetype
```

If you already cloned without `--recurse-submodules`, initialise them after the fact:

```bash
git submodule update --init --recursive
```

## Prerequisites

- **Docker** + Docker Compose (runs the whole backend stack)
- **[just](https://github.com/casey/just)** — the backend command runner
- **Node 26+** and **pnpm 11+** (see `frontend/package.json`) — for the frontend
- **[uv](https://docs.astral.sh/uv/)** — only needed if you run Python tooling outside Docker

You do **not** need a local Python 3.14 or Postgres install for normal work — the backend runs entirely in Docker.

## Standard local dev split

Archetype's runtime policy is: **backend in Docker, frontend on the host.**

### 1. Backend — Docker via `just`

From `api/`, create your env file and bring the stack up:

```bash
cd api
cp config/test.env config/.env
just up-bg          # start api, postgres, meilisearch, redis, celery, sipi, pgadmin
just migrate        # apply migrations (first run)
just setup-search-indexes   # create Meilisearch indexes
just sync-all-search-indexes
```

This exposes the API on `:8000`, Meilisearch on `:7700`, Postgres on `:5432`, Redis on `:6379`, and pgAdmin on `:5050`. API docs live at `http://localhost:8000/api/v1/docs`. Useful recipes: `just down`, `just restart-api`, `just shell`, `just pytest-focused`. Run `just` with no arguments to list them all.

### 2. Frontend — `pnpm dev` on the host

From `frontend/`, install dependencies, create your env file, and start the dev server:

```bash
cd frontend
pnpm install
cp .env.example .env
pnpm dev            # http://localhost:3000
```

The default `.env.example` already points at the local backend (`NEXT_PUBLIC_API_URL=http://localhost:8000`, `NEXT_PUBLIC_IIIF_UPSTREAM=http://localhost:1024`). See [Configuration](/deploy/configuration/) for the full list of environment variables and their meanings.

> Missing env vars fail startup — they are validated in `lib/env.ts` and `next.config.mjs`.

## Where things live

| Area | Path | Notes |
| --- | --- | --- |
| Backend apps | `api/apps/*` | Django feature apps; routes under `/api/v1/*` |
| Search registry | `api/apps/search/registry.py` | single source of truth for index types |
| Public site | `frontend/app/(site)/*` | App Router pages |
| Backoffice | `frontend/app/backoffice/*` | authoring / management UI |
| Data access | `frontend/lib/api-fetch.ts` | public fetch layer |
| Orchestration | `infrastructure/compose.yaml` | full-stack + nginx proxy |
